### 3.2 各场景详细分析

#### S1: Static (静态页面) - 100%
- **典型 URL**: news.ycombinator.com, docs.python.org
- **最佳工具**: web_fetch
- **特点**: 简单 HTML，无需 JS 渲染
- **表现**: 100% 成功，响应快速

#### S2: Anti-Scrape (反爬页面) - 75%
- **典型 URL**: medium.com, reddit.com, x.com
- **最佳工具**: browser (Relay)
- **特点**: 需要浏览器渲染，有反爬机制
- **问题**: 部分站点需要登录或验证码

#### S3: Platform (平台专属) - 85%
- **典型**: 知乎、微博、B站热点
- **最佳工具**: opencli
- **特点**: 结构化数据，API 优先
- **问题**: 部分平台接口变更频繁

#### S4: SPA (单页应用) - 70%
- **典型 URL**: react.dev, linear.app, notion.so
- **最佳工具**: browser
- **特点**: 需要等待 JS 渲染
- **问题**: 渲染时间长，部分应用需要登录

#### S5: Search (搜索查询) - 100%
- **典型查询**: "AI agents", "Python tutorial"
- **最佳工具**: search API (Tavily/Brave)
- **优化后**: Brave 优先策略，响应 -29%

#### S6: Realtime (实时数据) - 95%
- **典型**: 天气、汇率、热搜
- **最佳工具**: web_fetch (API) / opencli
- **特点**: 低延迟要求高

## 八、方法论总结

### 8.1 核心方法论

**"场景驱动 + 数据验证"优化法**

1. **场景定义**: 基于 URL/查询特征定义使用场景
2. **工具映射**: 为每个场景配置最优工具链
3. **Fallback 设计**: 设计降级策略保证可用性
4. **数据验证**: 通过实测验证策略有效性
5. **迭代优化**: 基于数据持续调整策略

### 8.2 关键经验

| 经验 | 说明 |
|------|------|
| 平台内容优先 CLI | opencli 比 browser 快 3-5 倍 |
| 搜索引擎重新排序 | Brave 优先比 Tavily 优先快 29% |
| 浏览器是最终兜底 | 但资源消耗大，应避免滥用 |
| 规则引擎足够 | 简单规则比 ML 更有效、可解释 |

### 8.3 可复用模式

```
┌─────────────────────────────────────────┐
│     工具选择决策树 (可复用)               │
├─────────────────────────────────────────┤
│  1. 检测输入类型 (URL vs Query)          │
│  2. 匹配场景规则 (Domain/Pattern)        │
│  3. 选择首选工具 (基于历史成功率)         │
│  4. 执行并验证结果                       │
│  5. 失败则 Fallback 到下一工具           │
│  6. 记录结果用于优化                      │
└─────────────────────────────────────────┘
```
