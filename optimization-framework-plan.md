## 零、项目缘起

### 0.1 研究背景

2026年3月，Andrej Karpathy 发布了 **autoresearch** 项目，一个月不到 GitHub Stars 突破 71k。这是一个让 AI Agent 自主进行 ML 实验的创新项目。

**autoresearch 核心理念**：
- 给 AI Agent 一个小型但真实的 LLM 训练环境
- Agent 自主修改代码、训练、评估、决策
- overnight 运行数百次实验
- 人类醒来后获得更好的模型

### 0.2 我们的分析

我们详细研究了 autoresearch 项目，撰写了分析文档：
- **分析文档**: `/workspace/jerry/research/autoresearch-inspiration.md`
| 设计思想 | 说明 |
|----------|------|
| Single file scope | Agent 只修改单一文件，保持可控 |
| Fixed time budget | 固定时间预算，结果可比 |
| Simplicity criterion | 简化优于复杂，删除代码是好结果 |
| program.md as skill | Markdown 文件作为 Agent 指令 |
| Git-based tracking | 结构化实验记录 |

### 0.3 决策与应用

**研究结论**：autoresearch 的方法论不仅适用于 ML 训练实验，其核心思想——
```
修改 → 执行 → 评估 → 决策 → 记录 → 重复
```
——可以应用到更广泛的优化场景。

**决策**：基于 autoresearch 方法论，设计并实现一个适合 OpenClaw 的优化测试框架，用于解决以下问题：
1. Skill vs 注意力优化
2. 长任务执行流程优化
3. 网页信息抓取工具优化

**重要说明**：autoresearch 本身是一个专门为 ML 训练设计的工具，不是现成可用的通用优化框架。我们需要**根据其思路自己设计并实现**适合 OpenClaw 的优化测试流程。

## 二、任务背景

### 2.1 为什么需要这个框架？

当前 OpenClaw 存在的优化需求：
| 问题 | 描述 | 优先级 |
|------|------|--------|
| Skill vs 注意力 | Skills 提醒触发不准确、用户注意力管理困难 | 高 |
| 长任务执行 | 任务执行流程稳定性和效果需要提升 | 高 |
| 网页信息抓取 | 不同场景需要最优工具组合 | 高 |

### 2.2 autoresearch 方法论借鉴

核心思想：
```
修改 → 执行 → 评估 → 决策 → 记录 → 重复
```

关键设计原则：
- **单一文件修改范围**：限定 Agent 操作边界
- **固定时间预算**：实验结果可比
- **简洁优于复杂**：删除代码也是好结果
- **结构化日志**：统一格式便于分析

### 8.3 执行策略：Subagent + 主Agent 总控

**核心原则**：专项任务由 Subagent 在独立上下文中执行，主 Agent 负责把关和总控。

**执行模式划分**：
| 任务类型 | 执行方式 | 理由 |
|----------|----------|------|
| 设计文档编写 | Subagent | 需专注思考，有明确规划指引 |
| 代码开发 | Subagent | 独立环境，可后台运行 |
| Review 验收 | 主 Agent | 需用户确认，交互性强 |
| 场景实验执行 | Subagent | 批量执行，不影响主 session |
| 总结文章输出 | Subagent | 专注写作，有素材积累 |

**Subagent spawn 规范**：
- 提供明确的 task 描述和目标
- 提供参考文档路径
- 提供输出要求和验收标准
- 完成后主 Agent 必须 Review 验收

### 8.4 执行原则

**Phase 1 执行时**（Subagent 模式）：
- 先完成设计文档，再开发代码
- 设计文档存放在 `design/optimization-framework/phase1-design/`
- 代码复杂度评估 → 决定是否需要 GitHub 仓库

**Phase 2 执行时**（主 Agent 验收）：
- Phase 1 完成后才能进入
- 设计测试用例 → 执行测试 → 验收确认
- 验收不通过则返回 Phase 1 修复

**Phase 3-5 执行时**（Subagent 模式）：
- Phase 2 验收通过后才能进入
- 每个场景独立执行
- 输出报告存放在对应 phase 目录

### 8.4 输出文件规范

**设计文档**：
- 存放于 `design/optimization-framework/phase{n}-xxx/`

- 包含：设计思路、架构图、决策记录
